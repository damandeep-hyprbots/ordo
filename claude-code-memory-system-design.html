Chapter 1

Architecture Overview

The memory system is a file-based knowledge store that persists across conversations. It has three scopes (personal, team, agent), four semantic types, and two ingestion paths (explicit writes + background extraction). Retrieval is relevance-ranked via a side-query to a fast model.

Why file-based, not a database? Files are transparent — users can cat, grep, and vi their memories. They're Git-friendly (team memories can be committed). They use the same tools the model already has (Read/Write/Edit). And they survive any process crash — no WAL, no corruption recovery needed.
Chapter 2

The Four Memory Types

Each memory has a semantic type that determines when it's saved, how it's structured, and how it should influence future behavior. Defined in memdir/memoryTypes.ts.

user Who is the user?

Role, goals, responsibilities, expertise, preferences. Always private — never shared with the team.

  • When saved: Learning the user's role, tech stack experience, communication preferences
  • Example: "Senior backend engineer, deep Go expertise, new to React frontend"
  • Purpose: Tailor explanations and collaboration style. A senior engineer gets architecture-level answers; a student gets step-by-step tutorials.
Why private? Personal context (skill level, working style) shouldn't leak to teammates. The model adapts its tone per person, not per team.

feedback How to work

Corrections and confirmations of approach. Body structure: rule → Why: → How to apply:

  • When saved: User says "don't do X" (correction) OR accepts a non-obvious approach (confirmation)
  • Example: "Integration tests must hit a real database, not mocks. Why: prior incident where mock/prod divergence masked a broken migration."
  • Key insight: Records from both failure and success. If only corrections are saved, the model grows overly cautious and drifts from validated approaches.
Why include "Why"? A bare rule ("don't mock DB") can't handle edge cases. Knowing the reason (mock/prod divergence) lets the model judge when the rule applies and when it doesn't — e.g., unit tests for pure functions can still use mocks.

project What's happening

Non-derivable facts about ongoing work, deadlines, decisions. Decays fast — always convert relative dates to absolute.

  • When saved: Learning who's doing what, why, or by when
  • Example: "Merge freeze begins 2026-03-05 for mobile release cut. Flag any non-critical PR work after that date."
  • Key rule: Must NOT be derivable from code or git history. If git log can tell you, don't save it.
Why absolute dates? "Next Thursday" is meaningless in a memory read 3 weeks later. Converting to "2026-03-05" ensures the memory remains interpretable as time passes.

reference Where to look

Pointers to external systems — not the data itself, just where to find it.

  • When saved: Learning about Linear projects, Grafana dashboards, Slack channels, wiki pages
  • Example: "Pipeline bugs tracked in Linear project 'INGEST'. Oncall latency dashboard: grafana.internal/d/api-latency."
  • Scope: Usually team-scoped — these references benefit everyone.
What NOT to save: Code patterns, architecture, file paths (derivable from reading code). Git history (use git log). Debugging solutions (the fix is in the code). CLAUDE.md content (already loaded). Ephemeral task details (use tasks instead). These exclusions apply even when the user explicitly asks — the model is instructed to push back and ask what was surprising or non-obvious.
Chapter 3

Storage & File Format

Memories are plain Markdown files with YAML frontmatter, organized in a per-project directory with a flat index file.

Why a separate index? MEMORY.md is always loaded into context (like CLAUDE.md). Individual memory files are loaded only when relevant via the retrieval pipeline. This keeps base context cost low — a project with 100 memories only pays for a 100-line index, not 100 full files.
Why per-project directories? The path uses the canonical Git root, so all worktrees of the same repo share memories. But different projects get separate directories — feedback about "how to test in project A" doesn't pollute project B.
// Path resolution priority (paths.ts) 1. CLAUDE_COWORK_MEMORY_PATH_OVERRIDE ← full override (Cowork mode) 2. settings.json → autoMemoryDirectory ← trusted sources only (NOT project settings!) 3. ~/.claude/projects/<sanitizePath(canonicalGitRoot)>/memory/ ← default
Security: Project settings excluded. A malicious repo could set autoMemoryDirectory: "~/.ssh" in its .claude/settings.json to gain write access to sensitive directories. Project-scoped settings are therefore intentionally blocked from configuring the memory path.
Chapter 4

Memory Lifecycle

A memory's journey: creation (two-step), update, staleness detection, verification-before-use, and deletion.

Why two-step creation? Writing the file and updating MEMORY.md are separate because the index must remain a curated summary, not an automatic mirror. The model can choose whether a memory is important enough to index. Orphaned files (in the directory but not in the index) are still discoverable via scanMemoryFiles() but won't appear in every conversation's context.
Why verify before recommending? A memory that says "function parseConfig() is in src/config.ts:42" is a claim about the past. The function may have been renamed, moved, or deleted. The model is explicitly instructed: "The memory says X exists" ≠ "X exists now." Always grep/check before acting on stale references.
Chapter 5

Retrieval Pipeline

Memories are surfaced via three complementary mechanisms: always-on index injection, relevance-ranked file selection, and on-demand access by the model.

Why three mechanisms? Each serves a different need. The always-on index gives the model awareness of what exists (cheap — just file names). Relevance prefetch proactively loads the right memories for the current task (no model action needed). On-demand access is the fallback for explicit user requests or edge cases the prefetch missed.
Why a Sonnet side-query? Semantic relevance can't be solved by keyword matching — "should I mock the database?" is related to a memory about "integration test policy" despite no shared keywords. A small, fast model (Sonnet) reads the frontmatter descriptions and selects the most relevant files. This runs in parallel with the main model's response, adding zero latency to the user experience.
Why a 60KB session cap? Without it, long conversations could accumulate hundreds of KB of surfaced memories, crowding out actual conversation content from the context window. Once 60KB is reached, the prefetch stops — the model can still manually read files if needed.
Chapter 6

Background Extraction Agent

At the end of each turn, a forked agent silently reviews the conversation and saves memories the main agent missed — sharing the parent's prompt cache for efficiency.

Why a forked agent? The main model is focused on answering the user's question — asking it to also extract memories would split its attention and add latency. A runForkedAgent() shares the parent's prompt cache (so no re-encoding cost) but runs independently in the background. The user sees their answer immediately; memories are saved silently afterward.
Why skip if main already wrote? If the main agent proactively saved a memory (e.g., user said "remember this"), the extraction agent has nothing to add. The hasMemoryWritesSince() check prevents duplicate work and avoids conflicting writes.
Chapter 7

Session Memory (Intra-Conversation)

A separate, session-scoped system that maintains a structured summary of the current conversation — surviving context compaction but NOT persisting across sessions. This is distinct from the persistent memory system.

Why separate from persistent memory? Session memory tracks ephemeral state: "which files have I touched," "what errors did I hit," "what's my current approach." This is invaluable when context is compacted mid-conversation (the model loses old messages but keeps the summary), but useless in future conversations. Persistent memory is the opposite — designed for cross-conversation knowledge.
Why structured sections? A free-form summary would be hard to maintain incrementally. Fixed sections (Task, Files, Errors, Learnings, Worklog) let the update agent surgically modify one section without rewriting the entire summary. Each section has a 2,000-token cap to prevent any single aspect from dominating.
Chapter 8

Team Memory Sync

A shared memory layer synced via Anthropic's API. Server wins on pull, delta-only push, with secret scanning to prevent credential leaks.

Why server-wins on pull? Team memory is a shared resource. If two people edit the same file, the last pusher wins on the server, and everyone else gets that version on pull. This is simpler than merge conflict resolution and matches the expected usage pattern: team memories are reference docs, not constantly-edited collaboration documents.
Security: Symlink traversal prevention. A two-pass validation checks: (1) string-level path containment after path.resolve(), (2) real filesystem containment via realpathDeepestExisting() to catch symlink escape attacks. Also blocks null bytes, URL-encoded traversals, and Unicode normalization attacks (fullwidth ．．／).
Chapter 9

Agent Memory

Custom agents (created via the Agent creation wizard) get their own isolated memory directories, scoped by agent type and visibility level.

user scope ~/.claude/agent-memory/<type>/

Memories persist across all projects for this user. Recommended for user-scoped agents. Guidance: "keep learnings general since they apply across all projects."

project scope .claude/agent-memory/<type>/

Lives inside the project directory, shared via version control. Recommended for project-scoped agents. Guidance: "tailor your memories to this project."

local scope .claude/agent-memory-local/<type>/

Project-specific but NOT in version control. For machine-specific configuration that shouldn't be shared with teammates.

none No persistence

Agent starts fresh every invocation. Used for stateless, deterministic agents where memory would add unwanted variability.

Why separate from the main memory? An agent (e.g., a code reviewer) has different context than the main Claude conversation. A code reviewer's memory about "this team prefers explicit error handling over panics" should only be surfaced when the code reviewer runs, not during general conversation. Agent memory isolation prevents cross-contamination.
Chapter 10

Security Model

Memory writes happen in sensitive directories (~/.claude/). Multiple layers prevent path traversal, credential leaks, and malicious project configurations.

Path Validation validateMemoryPath()

Rejects: relative paths, root/near-root paths, Windows drive roots, UNC paths (\\), null bytes. Ensures all writes stay within the designated memory directory.

Symlink Defense Two-pass containment

Pass 1: path.resolve() normalizes .. segments. Pass 2: realpathDeepestExisting() resolves symlinks on the real filesystem. Both must be contained within the memory dir. Prevents symlink escape attacks (PSR M22186).

Secret Scanning gitleaks rules

Before team memory push, each file is scanned for secrets (API keys, tokens, passwords). Files with detected secrets are silently skipped with a warning — never uploaded to the server.

Project Settings Blocked autoMemoryDirectory

A malicious repo's .claude/settings.json could set autoMemoryDirectory: "~/.ssh". Project settings are therefore intentionally excluded from configuring the memory path — only user/policy/local settings are trusted.

Unicode Normalization sanitizePathKey()

Blocks fullwidth Unicode traversals (．．／), URL-encoded sequences (%2e%2e%2f), backslashes, absolute paths, and null bytes in team memory path keys.

Dangerous Dir Carve-out Filesystem permissions

~/.claude/ is in DANGEROUS_DIRECTORIES. An explicit carve-out allows memory writes only when using the default path — env var overrides get no special treatment and must supply their own allow rules.
