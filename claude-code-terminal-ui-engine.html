Chapter 1

Custom Rendering Pipeline

Upstream Ink builds strings and reprints them. This fork maintains a 2D cell grid with typed pools, double-buffering, and an incremental diff engine that emits only changed cells.

Why typed pools (StylePool, CharPool)? Interning styles and characters by ID means identical cells across frames share the same pool entry. The diff engine compares pool IDs (integers) rather than style objects — O(1) equality check per cell, enabling fast full-screen diffs without string comparison.
Why DECSTBM hardware scroll? When content scrolls up, DECSTBM tells the terminal to scroll its own framebuffer — the GPU/display controller moves pixels without the CPU repainting every line. On large terminals with long message histories, this can be the difference between smooth 60fps and visible tearing.
Chapter 2

Virtual Message List

Long conversations can have hundreds of messages. Mounting all of them creates React churn and kills performance. The virtual scroll system mounts only what's visible ± an overscan buffer.

Why SCROLL_QUANTUM=40 rows? A mouse wheel notch produces 5+ scroll events. Without quantization, each tick triggers a React re-render. By snapping scrollTop to 40-row bins, React only sees a new state when the visible range actually changes — reducing scroll renders by ~95%.
Why DEFAULT_ESTIMATE=3 (intentionally low)? Overestimating height creates blank space below items before measurement. Underestimating causes slight layout jump but no visible gap. Users perceive layout-jumps as natural content reveal; blank space looks like a bug.
Chapter 3

Streaming Text & Ratchet Component

As tokens stream in, text appears progressively. The Ratchet component ensures message height never shrinks — preventing layout shift that makes content feel unstable.

streamingText Live Rendering

streamingText: string | null rendered as a separate Box appended to message list. Streaming tool uses with pending state get synthetic messages via syntheticStreamingToolUseMessages. inProgressToolUseIDs set gates spinner vs rendered output.

Ratchet Height Lock

Measures element height via measureElement on every useLayoutEffect. Maintains a monotonically increasing minHeight. Once a message renders at height N, it never collapses smaller — even as streaming continues and text reflows.

Viewport Aware Pause Off-Screen

Ratchet pauses height-locking when isVisible=false via useTerminalViewport. Off-screen messages don't hold scroll height unnecessarily — only visible messages maintain their ratcheted minimum.

Why ratchet instead of fixed heights? Token streaming can cause text to reflow as it grows (line wraps at different points with more text). Without ratcheting, each new token could collapse previously expanded content, causing the viewport to jump. The ratchet trades a tiny bit of extra vertical space for visual stability.
Chapter 4

Focus Management System

A DOM-like focus manager with a history stack, tab order traversal, and discrete React dispatch for tear-free focus transitions.

FocusManager Architecture

  • Stored on root DOMElement (like document)
  • Focus stack: max 32 entries, deduped
  • handleNodeRemoved restores to most recent still-mounted element
  • Custom FocusEvent class with relatedTarget

Tab Order Traversal

collectTabbable() walks the element tree, collects all nodes with tabIndex ≥ 0. focusNext() / focusPrevious() wrap modularly. Same model as browser tab navigation.

Dialog Lock enable/disable

enable() / disable() globally gate focus changes. Used when permission dialogs appear — disabling focus prevents the prompt input from stealing focus back while the dialog awaits a decision.

Discrete Dispatch Tear-Free

Focus events dispatch via React reconciler's dispatchDiscrete() — same mechanism as browser click handlers. Ensures focus state updates are batched with their triggering event, preventing torn intermediate states.

Chapter 5

Full Vim Mode

A complete Vim emulator inside the terminal prompt — INSERT/NORMAL modes, all major motions, operators, text objects, find, and dot-repeat.

// Arrow key remapping in NORMAL mode Arrow keys → hjkl (motion-expecting states) Backspace → h (move left) Delete → x (delete char) // INSERT→NORMAL cursor adjustment on Escape: cursorOffset = Math.max(0, cursorOffset - 1) // Matches real vim behavior: cursor shifts left on Esc
Why implement Vim in a terminal prompt? Developers who use Vim spend years building muscle memory. A 40-character prompt edit that takes 3 seconds in INSERT mode takes 0.5 seconds with Vim motions (cw to change a word, $ to jump to end). For a tool used hundreds of times per day, this matters.
Chapter 6

Image Paste Support

Handles clipboard images, Finder drag-and-drop of multiple files, screenshot paths, and large bracketed paste chunking.

Bracketed Paste Detection

Terminal signals paste via \x1b[200~ ... \x1b[201~. isPasted flag set by keypress parser. Previous approach used competing stdin.on('data') listener that caused dropped characters.

Path Detection Multi-Image

Pasted text split on / (?=\/|[A-Za-z]:\\)/ (space before Unix / or Windows C:\) then on \n. Each segment tested with isImageFilePath. Handles Finder multi-image drag as space-separated paths.

Clipboard Image macOS Only

Empty bracketed paste (input.length === 0) → calls getImageFromClipboard() debounced 50ms. Also triggered for temp screenshot paths like TemporaryItems/.../screencaptureui/.../Screenshot that no longer exist on disk.

Chunk Aggregation Large Pastes

Large pastes arrive in multiple stdin chunks. A 100ms setTimeout aggregates them. pastePendingRef.current (synchronous ref) prevents a subsequent keystroke in the same React batch from bypassing the accumulator.

Chapter 7

Terminal Capability Detection

Uses a DA1 (Device Attributes) sentinel as an ordering barrier — all capability queries fire in parallel, then the DA1 response signals "all previous responses received". No timeouts needed.

Why OSC 11 for theme? $COLORFGBG is unreliable (many terminals don't set it, or set it incorrectly). OSC 11 queries the terminal's actual background color and lets Claude Code reliably distinguish light-on-dark from dark-on-light terminals — no manual theme selection needed in most cases.
Chapter 8

Animation System

A shared clock with viewport culling, terminal focus throttling, per-character shimmer, and stall detection that fades the spinner red after 3 seconds of silence.

Why viewport culling for animations? A conversation with 200 messages could have 200 active animation ticks per frame if every spinner ran regardless of visibility. Culling to the visible range + overscan keeps the animation budget O(screen height) rather than O(message count).
Chapter 9

Theme System

6 named palettes, 60+ semantic color slots, and an OSC 11 watcher for automatic light/dark detection.

6 Palettes Named Themes

  • dark / light — default
  • dark-daltonized / light-daltonized — color blindness
  • dark-ansi / light-ansi — 16-color terminals only

Semantic Slots 60+ Colors

Named by purpose, not value: claude, claudeShimmer, permission, planMode, diffAdded/diffRemoved/diffAddedWord/diffRemovedWord, 8 subagent colors (_FOR_SUBAGENTS_ONLY), 7 rainbow colors + shimmers.

Auto Detection OSC 11 Watcher

watchSystemTheme() queries the terminal's actual background color via OSC 11. Seeds from $COLORFGBG initially, then corrects with the real value. Preview mode (setPreviewTheme) shows live previews without saving.

Chapter 10

Search & Highlight

Two-layer highlighting: all matches get inverted, the current match gets yellow+bold+underline — implemented at both the rendered-screen and off-screen-buffer levels.

Screen-Space applySearchHighlight

Post-render, walks every row of the 2D screen buffer. Builds codeUnitToCell[] map handling wide chars, SpacerTail/SpacerHead, noSelect gutters. Non-overlapping matches get SGR 7 (inversion) via stylePool.withInverse.

Current Match applyPositionedHighlight

The focused match gets yellow+bold+underline via applyPositionedHighlight, writing on top of the inversion. Uses message-relative positions stored in Ink.searchPositions, adjusted by scroll offset.

Off-Screen Scan renderToScreen

renderToScreen(el, width) renders a React element to an isolated Screen buffer (~1-3ms). scanPositions() finds all match positions in the buffer. Enables searching messages that aren't currently mounted in the virtual scroll.

Notification System

Terminal-native notifications via OSC escape sequences — different protocols for different terminal emulators.

MethodProtocolFormat

Why NOT wrap BEL in multiplexer passthrough? tmux needs to see the raw BEL character to trigger its bell-action (which can flash the pane border, run a command, etc.). The OSC multiplexer wrapper (DCS tmux; ... ST) passes through OSC sequences but may swallow raw BEL. So BEL is sent unwrapped intentionally.

Tab Status OSC 21337

Three presets: idle (rgb(0,215,95) "Idle"), busy (rgb(255,149,0) "Working…"), waiting (rgb(95,135,255) "Waiting"). Currently Anthropic-internal rollout only.

Idle Notify useNotifyAfterTimeout

Sends desktop notification when user has been idle >6s and a response completes. Uses a polling interval (not single setTimeout) to catch tasks completing after an already-idle user. Interaction tracked in processKeysInBatch.
