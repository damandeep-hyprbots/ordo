Claude Code — Bridge / Remote Control

A deep dive into the system that connects a local Claude Code CLI to Anthropic's CCR (Claude Code Remote) infrastructure, enabling claude.ai to dispatch prompts to your machine and stream responses back in real time.

Section 1

Bridge Mode Overview

Bridge Mode (marketed as "Remote Control") lets a user's claude.ai browser session dispatch prompts to a local Claude Code CLI running on their development machine. The local CLI executes tools, reads files, runs commands, and streams results back to the web UI.

Why OAuth only? The bridge authenticates to CCR using the claude.ai OAuth token. This excludes Bedrock, Vertex, Foundry, API keys, and Console API logins because none of those credentials carry the OAuth token that CCR requires. The function isClaudeAISubscriber() enforces this gate at runtime.

Core What Bridge Does

Connects your local Claude Code to Anthropic's cloud so that a claude.ai web session can send prompts to your machine and receive tool outputs, file reads, and command results streamed back in real time.

Build DCE Gating

feature('BRIDGE_MODE') is a Bun bundler dead-code elimination gate. In external builds, the entire bridge module tree is tree-shaken out -- no code, no string literals, no GrowthBook flag names remain in the bundle.

Gate GrowthBook Flag

At runtime, tengu_ccr_bridge controls whether bridge is enabled for a given organization. The blocking variant checkGate_CACHED_OR_BLOCKING ensures stale disk-cache misses don't unfairly deny access.

Section 2

Bridge Architecture

bridgeMain.ts is a full polling/spawning daemon. It polls CCR for work, spawns child sessions to handle prompts, manages heartbeats and token refresh, and coordinates up to 32 concurrent sessions per environment.

Why 32 concurrent sessions? SPAWN_SESSIONS_DEFAULT = 32 bounds the number of active child processes the daemon manages. Each session is a separate Claude Code process with its own conversation, tool execution, and working directory. The limit prevents runaway resource consumption while allowing meaningful parallelism for power users working across multiple contexts.

Poll Work Dispatch

The bridge daemon continuously polls GET /bridge for new work items. Each work item contains a session ID, an ingress JWT, and optional metadata. Backoff config: 2s initial, 2min cap, 10min give-up.

Spawn Session Management

Work items are handed to createSessionSpawner() which forks child claude processes. Each child gets an --sdk-url pointing to the session's transport endpoint and the ingress JWT for authentication.

Health Heartbeat + Token Refresh

Active work items are heartbeated to CCR. 401/403 responses trigger reconnectSession (re-queues the work for fresh JWT delivery). A separate scheduler handles OAuth token refresh before expiry.

v1 vs v2 Transport Split

  • v1: HybridTransport (WebSocket reads + HTTP POST writes to Session-Ingress)
  • v2: SSETransport (reads) + CCRClient (writes to /worker/*)
  • Gated by tengu_bridge_repl_v2 GrowthBook flag

Mirror CCR_MIRROR Mode

When enabled, every local session spawns an outbound-only shadow session. The shadow forwards events to CCR but never receives inbound prompts. The outboundOnly flag on v2 transport skips SSE stream entirely.

Auto CCR_AUTO_CONNECT

Makes all sessions connect to CCR by default (opt-out rollout). Controlled by tengu_cobalt_harbor GrowthBook gate. Users can explicitly set remoteControlAtStartup=false in config to opt out.

Section 3

Transport Layer (HybridTransport)

The primary bridge transport combines a WebSocket (or SSE stream) for receiving data with HTTP POST for sending. The v2 path separates these concerns further with SSETransport + CCRClient, while both versions share the same ReplBridgeTransport interface.

Why serialize writes? Bridge mode fires writes via void transport.write() (fire-and-forget). Without serialization, concurrent POSTs cause concurrent Firestore writes to the same document, leading to collisions, retry storms, and pages to the on-call team. The SerialBatchEventUploader ensures at most one POST is in-flight at any time.
Why a 45-second liveness timeout? The server sends keepalive comments (:keepalive) every 15 seconds. If no SSE frame arrives within 3x that interval (45 seconds), the connection is presumed dead. This avoids waiting indefinitely on a silently-broken TCP connection while giving enough headroom for normal network jitter.
// SSETransport configuration constants const RECONNECT_BASE_DELAY_MS = 1000 const RECONNECT_MAX_DELAY_MS = 30_000 const RECONNECT_GIVE_UP_MS = 600_000 // 10 minutes const LIVENESS_TIMEOUT_MS = 45_000 // 3x keepalive interval // HybridTransport POST configuration const BATCH_FLUSH_INTERVAL_MS = 100 // buffer stream events 100ms const POST_TIMEOUT_MS = 15_000 // per-attempt timeout const CLOSE_GRACE_MS = 3000 // drain queue on close

Read WebSocket / SSE

v1 uses a WebSocket for receiving inbound events; v2 uses an SSE stream at /worker/events/stream. Both handle reconnection with exponential backoff. SSE carries sequence numbers for resume-from-last.

Write Serialized POST

Both versions serialize writes through a batch uploader. Stream events accumulate for 100ms before flushing. At most one HTTP POST is in-flight. Failed POSTs retry with exponential backoff (500ms base, 8s cap, 1s jitter).

Swap Transport Hot-Swap

When swapping transports, getLastSequenceNum() captures the SSE high-water mark so the new transport resumes from where the old one left off. Without this, the server replays the entire session history from sequence 0.

Section 4

SessionsWebSocket

A dedicated WebSocket client for subscribing to CCR session streams. Used by the claude.ai frontend (and remote session manager) to receive real-time SDK messages from active sessions.

Why allow retries on 4001? During message history compaction, the server may briefly consider the session stale while the CLI worker is busy with the compaction API call and not emitting events. The 4001 code can be transient in this window, so up to 3 retries (with escalating delay) let the client recover without permanently giving up.
// SessionsWebSocket constants const RECONNECT_DELAY_MS = 2000 const MAX_RECONNECT_ATTEMPTS = 5 const PING_INTERVAL_MS = 30000 const MAX_SESSION_NOT_FOUND_RETRIES = 3 // Permanent close codes (stop reconnecting immediately) const PERMANENT_CLOSE_CODES = new Set([ 4003, // unauthorized ]) // WebSocket URL pattern wss://api.anthropic.com/v1/sessions/ws/{sessionId}/subscribe?organization_uuid=...

Auth Header-Based

Authentication is sent via HTTP headers on the WebSocket upgrade request: Authorization: Bearer {token} plus anthropic-version: 2023-06-01. Bun and Node/ws handle this differently in implementation but the protocol is identical.

Close Graceful Shutdown

On close, event handlers are detached, the ping interval is cleared, and any pending reconnect timer is cancelled. The reconnect() method resets all retry counters for a clean restart after stale subscriptions.

Control Bidirectional

The WebSocket is bidirectional: the server streams SDKMessages, while the client can send back SDKControlResponse (permission grants) and SDKControlRequest (interrupts) with unique request IDs.

Section 5

Real-Time Protocols Summary

Claude Code uses multiple real-time transport protocols, each optimized for a different communication pattern. Here is a comprehensive map of every WebSocket and streaming connection in the system.

Protocol Transport Direction Purpose Auth Reconnect
SessionsWebSocket WebSocket (WSS) Bidirectional CCR session streaming: receive SDK messages, send control responses/requests OAuth token in headers 5 attempts, 2s delay; 3 retries for 4001
SSETransport Server-Sent Events (HTTP) Server → Client (read) Bridge v2 ingress: receive inbound events from CCR worker endpoint JWT (session ingress token) Exp backoff 1s-30s, 10min budget
HybridTransport WebSocket + HTTP POST WS: read; POST: write Bridge v1: primary transport combining real-time reads with serialized writes Session ingress token (env var) Inherited from WebSocketTransport
VoiceStream WebSocket Bidirectional Speech-to-text audio streaming for voice mode (separate from MCP/CCR) Per-service auth Feature-gated (VOICE_MODE)
mcpWebSocketTransport WebSocket Bidirectional MCP protocol over WebSocket for remote MCP servers (alternative to stdio) Per-server config MCP client managed
Why so many transports? Each protocol serves a distinct use case with different reliability/latency tradeoffs. SSE is ideal for unidirectional server pushes with built-in reconnection semantics (Last-Event-ID). WebSockets provide full-duplex communication for interactive protocols. HTTP POST offers reliable, retriable writes with batch semantics. The HybridTransport combines WebSocket reads with POST writes because the bridge write path needs stronger delivery guarantees than WebSocket's fire-and-forget semantics.
Section 6

Dead Code Elimination

The Bun bundler's feature() macro enables compile-time dead code elimination. In external builds, the entire bridge module tree -- code, string literals, and GrowthBook flag names -- is tree-shaken out of the bundle. This requires careful coding discipline.

Critical discipline: The negative pattern if (!feature('X')) return does NOT fully eliminate string literals from the bundle. The bundler sees if (!false) return which early-returns before the call, but the string argument to downstream functions may still survive as a reference in the compiled output. Always use the positive ternary: feature('X') ? realCode : false.
// bridgeEnabled.ts -- every bridge entry point uses this pattern export function isBridgeEnabled(): boolean { // Positive ternary pattern -- see docs/feature-gating.md return feature('BRIDGE_MODE') ? isClaudeAISubscriber() && getFeatureValue_CACHED_MAY_BE_STALE('tengu_ccr_bridge', false) : false } export function isEnvLessBridgeEnabled(): boolean { return feature('BRIDGE_MODE') ? getFeatureValue_CACHED_MAY_BE_STALE('tengu_bridge_repl_v2', false) : false } export function isCcrMirrorEnabled(): boolean { return feature('CCR_MIRROR') ? isEnvTruthy(process.env.CLAUDE_CODE_CCR_MIRROR) || getFeatureValue_CACHED_MAY_BE_STALE('tengu_ccr_mirror', false) : false } export function getCcrAutoConnectDefault(): boolean { return feature('CCR_AUTO_CONNECT') ? getFeatureValue_CACHED_MAY_BE_STALE('tengu_cobalt_harbor', false) : false }
Why is string elimination important? GrowthBook flag names like tengu_ccr_bridge are internal identifiers that reveal feature rollout strategy, targeting rules, and internal naming conventions. If these strings survive in the external bundle, they leak operational details. The excluded-strings.txt enforcement ensures these never appear in production external builds.

Gate feature() Macro

The feature('FLAG_NAME') call from bun:bundle is resolved at compile time. For external builds, bridge flags evaluate to false, causing the bundler's constant-folding pass to eliminate the dead branch entirely.

Tree Module Tree Shaking

When all call sites of a module are eliminated by DCE, the module itself becomes unreachable. Bun's tree-shaker removes the entire import chain: bridgeMain.ts, bridgeApi.ts, replBridge.ts, and all their transitive dependencies.

Verify excluded-strings.txt

A build-time check scans the external bundle for strings that must not appear (GrowthBook flag names, internal URLs, etc). If any survive, the build fails. This is the safety net for the positive-ternary discipline.

Flags Build-Time Constants

  • BRIDGE_MODE -- entire bridge/remote control system
  • CCR_AUTO_CONNECT -- auto-connect default
  • CCR_MIRROR -- outbound-only shadow sessions
  • KAIROS -- assistant mode
  • COORDINATOR_MODE -- multi-agent
  • VOICE_MODE -- voice features
