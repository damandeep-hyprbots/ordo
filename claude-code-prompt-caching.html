Prompt Caching & Anti-Distillation

A deep dive into how Claude Code minimizes API latency and cost through intelligent prompt caching, detects cache breaks, manages effort levels, and protects model behavior from distillation attacks.

Overview Cache Breakpoints System Prompt Blocks Cache TTL Break Detection Effort Levels Fake Tools Connector Text Model Fallback API Providers
Section 1

Why Prompt Caching Matters

Every API call sends the full system prompt, tool definitions, and conversation history. Without caching, the model re-encodes all of this from scratch on every turn -- a massive waste of compute, time, and money.

Why? A typical Claude Code session accumulates 50-200K tokens of context across system prompt, tool definitions, CLAUDE.md files, and conversation history. Re-encoding all of that on every turn would cost 10x more and add 1-2 seconds of latency. Cache breakpoints tell the API: "everything before this marker is identical to last time -- skip straight to the KV cache."

Cost Token Savings

Cached input tokens are billed at 10% of the standard rate. For long conversations with large system prompts, this compounds dramatically.

Latency Time to First Token

The model skips the encoding pass for cached content. The KV cache lookup is near-instant compared to full forward passes through the transformer.

Stability Session Consistency

Cache TTLs and eligibility are latched at session start. Mid-session state changes (overage flips, feature flag updates) cannot bust the cache.

Section 2

Cache Breakpoint Placement

The addCacheBreakpoints() function places exactly ONE cache_control marker per request. The placement depends on whether the request is a fire-and-forget fork or a normal request.

Why only one marker? Mycro's turn-to-turn KV eviction (page_manager/index.rs: Index::insert) frees local-attention KV pages at any cached prefix position NOT in cache_store_int_token_boundaries. With two markers, the second-to-last position would be protected and its pages would survive an extra turn even though nothing will ever resume from there. With one marker, they are freed immediately. For fork requests, shifting to the penultimate message means the write is a no-op merge (entry already exists in KVCC) and the fork does not leave its own tail in the cache.
const markerIndex = skipCacheWrite ? messages.length - 2 // penultimate: last shared-prefix point : messages.length - 1 // last message: normal cache write const result = messages.map((msg, index) => { const addCache = index === markerIndex // Each message converter applies cache_control only when addCache=true return msg.type === 'user' ? userMessageToMessageParam(msg, addCache, enablePromptCaching, querySource) : assistantMessageToMessageParam(msg, addCache, enablePromptCaching, querySource) })
Section 3

System Prompt Cache Blocks

splitSysPromptPrefix() segments the system prompt into cacheable blocks with different scopes depending on the deployment mode. The goal is to maximize cross-user cache hits for static content while keeping dynamic content per-organization.

Why? Global-scope caching shares the KV cache for identical static content across all users on the same model -- a huge multiplier when millions of CLI sessions share the same base system prompt. But dynamic content (CLAUDE.md, MCP tool context, user settings) must be scoped to the org or left uncached, because it differs between users. When MCP tools are present, the system prompt boundary cannot reliably separate static from dynamic content, so everything falls back to org scope.
Section 4

Cache TTL Management

getCacheControl() returns the cache_control object attached to each cacheable block. The TTL determines how long the server-side KV cache entry persists.

function getCacheControl({ scope, querySource }) { return { type: 'ephemeral', ...(should1hCacheTTL(querySource) && { ttl: '1h' }), ...(scope === 'global' && { scope }), } } // Result for eligible users: { type: 'ephemeral', ttl: '1h' } // 1-hour TTL // Result for non-eligible users: { type: 'ephemeral' } // 5-minute default TTL

Eligibility Who Gets 1h TTL?

Anthropic employees (USER_TYPE=ant), Claude AI subscribers who are not over their rate limit, or 3P Bedrock users who opt in via ENABLE_PROMPT_CACHING_1H_BEDROCK.

Latching Session Stability

setPromptCache1hEligible() latches the eligibility decision into bootstrap state on the first check. If a user goes into overage mid-session, their TTL does not flip from 1h to 5m. The cache stays warm.

Allowlist Source Filtering

A GrowthBook allowlist controls which query sources get 1h TTL. Patterns support trailing * for prefix matching. The allowlist is also cached in state for session stability.

Why latch eligibility? If the TTL changed from '1h' to nothing (5-minute default) mid-session, the entire cache_control block changes. Because cache_control is part of the hashed prompt state, this would bust the server-side cache, forcing a full re-encode of ~20K+ tokens. By latching the decision at session start, we guarantee the TTL field stays stable for the entire session.
Section 5

Cache Break Detection

promptCacheBreakDetection.ts is a two-phase system that detects when the prompt cache was unexpectedly invalidated and reports exactly what changed.

Why two phases? Phase 1 runs before the API call when we still have access to the full prompt state. Phase 2 runs after the response arrives, when we can see the actual cache token counts. Separating them avoids hashing the prompt twice and lets us correlate "what changed" (phase 1) with "did the cache actually break" (phase 2). The 5% threshold with a 2000-token minimum filters out noise from normal variation.

Tracking Source Isolation

Each query source (main thread, subagents, SDK) is tracked independently via previousStateBySource. Compact shares tracking with main thread since they share the same server-side cache key. Map is capped at 10 entries to prevent memory leaks from many subagents.

Per-Tool Hashing Precision Diffs

When tools change but the count stays the same, per-tool hashes identify exactly which tool's schema was modified. AgentTool and SkillTool are common culprits because they embed dynamic agent/command lists in their descriptions.

TTL Detection Not a Bug

If no client-side changes are detected but the time gap exceeds 5 minutes (or 1 hour for 1h-eligible users), the break is attributed to TTL expiration rather than a client bug. ~90% of "no changes" breaks are server-side routing/eviction.

Section 6

Effort Levels

Effort controls how deeply the model thinks before responding. It flows through a priority chain from environment variables to model-specific defaults, and is sent as part of the API request's output_config.

Why? Effort levels let users trade thinking depth for speed and rate limits. The 'medium' default for Opus 4.6 Pro subscribers ("tengu_grey_step2") balances intelligence with rate limit headroom -- users can trigger deeper thinking with the "ultrathink" feature when needed, which bumps to 'high'. Numeric effort values (ant-only) go through a separate anthropic_internal.effort_override path and are never persisted to user settings.
Section 7

Anti-Distillation: Fake Tools

To prevent competitors from distilling Claude's behavior by observing tool definitions in API traffic, Claude Code can request the API to inject fake/dummy tool definitions into the prompt.

Why? If an adversary intercepts API requests and observes the exact tool definitions Claude was given, they could use those definitions to create a distilled training dataset. By injecting fake tools server-side, the model's behavior is shaped by tools the adversary cannot see in transit. Even with the real tool definitions, the distilled model would not replicate Claude's exact decision-making because it would be missing the fake tool context that influenced the original responses.
// Anti-distillation: send fake_tools opt-in for 1P CLI only if ( feature('ANTI_DISTILLATION_CC') ? process.env.CLAUDE_CODE_ENTRYPOINT === 'cli' && shouldIncludeFirstPartyOnlyBetas() && getFeatureValue_CACHED_MAY_BE_STALE('tengu_anti_distill_fake_tool_injection', false) : false ) { result.anti_distillation = ['fake_tools'] }
Section 8

Anti-Distillation: Connector Text Summarization

A second anti-distillation layer operates on the assistant's free-text responses between tool calls. The server summarizes connector text and returns a cryptographic signature, preventing raw assistant reasoning from being observed in transit.

Why? The assistant's free-text reasoning between tool calls reveals Claude's planning strategy, chain-of-thought patterns, and decision-making heuristics. An adversary capturing API traffic could use this text directly as training data. By summarizing it server-side and only transmitting the summary, the raw reasoning is never observable in transit. The cryptographic signature ensures the original text can be faithfully restored on subsequent turns without the client needing to store or retransmit it.
Section 9

Model Fallback (FallbackTriggeredError)

When the primary model is overloaded, Claude Code automatically falls back to a secondary model after repeated 529 errors, keeping the user productive instead of stuck in a retry loop.

Why? During high-demand periods, Opus 4.6 may return 529 (overloaded) errors. Rather than leaving the user waiting through exponential backoff that may never succeed, after 3 consecutive 529s the system automatically switches to the fallback model. The current turn's assistant messages are cleared before retry because they may contain thinking signatures from the original model that would be invalid on the fallback. The user is warned so they understand any quality difference.
Section 10

API Provider Support

Claude Code supports multiple API providers with a unified retry and configuration layer. Each provider has different capabilities, authentication flows, and cache behaviors.

1P Anthropic Direct

Primary provider. Full feature support including global cache scope, anti-distillation, 1h TTL, connector text summarization, and all beta headers. Uses @anthropic-ai/sdk directly.

AWS Bedrock

3P provider with org-scoped caching. Supports 1h TTL via ENABLE_PROMPT_CACHING_1H_BEDROCK env opt-in. Users manage their own billing. No global cache scope or anti-distillation features.

GCP Vertex AI

3P provider with credential refresh handling. Vertex-specific auth errors (401, credential failures) trigger fresh client creation on retry. Standard org-scoped caching.

// withRetry: exponential backoff for 429/529 errors for (let attempt = 1; attempt <= maxRetries + 1; attempt++) { if (options.signal?.aborted) throw new APIUserAbortError() try { // Fresh client on first attempt or after auth errors const result = await operation(client, attempt, retryContext) return result } catch (error) { // 529 tracking for model fallback if (is529Error(error)) { consecutive529Errors++ if (consecutive529Errors >= MAX_529_RETRIES) { throw new FallbackTriggeredError(options.model, options.fallbackModel) } } // Exponential backoff with jitter... } } // Extra body params from environment // CLAUDE_CODE_EXTRA_BODY env var parsed and merged into request getExtraBodyParams() // reads CLAUDE_CODE_EXTRA_BODY // Task budget beta for long-running operations configureTaskBudgetParams() // sets output_config.task_budget
Why? Supporting multiple providers gives enterprise users deployment flexibility (data residency with Bedrock/Vertex) while maintaining a unified caching and retry layer. The withRetry wrapper handles provider-specific quirks (Bedrock auth errors vs Vertex credential refresh vs 1P OAuth revocation) behind a single async generator interface, yielding system error messages for the UI to display during retries.

Retry withRetry Wrapper

Async generator that yields SystemAPIErrorMessage events during retries so the UI can show "Retrying in Xs..." messages. Handles 429 rate limits, 529 overloaded, auth errors, and stale connections (ECONNRESET/EPIPE) with fresh client creation.

Config Extra Body Params

CLAUDE_CODE_EXTRA_BODY env var lets advanced users inject arbitrary fields into the API request body. Hashed by cache break detection to track when these params change and bust the prompt cache.

Budget Task Budget Beta

configureTaskBudgetParams sets output_config.task_budget for long-running agent operations, giving the model a token budget hint for planning multi-step tasks.
