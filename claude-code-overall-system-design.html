Chapter 1

High-Level Overview

Claude Code is a terminal-native AI assistant. At its heart it is an async-generator pipeline that streams messages between the user, the Claude API, and a rich tool system — all rendered in a custom React-based TUI.

Why this architecture? Claude Code needs to handle long-running, multi-turn AI conversations with tool execution — all inside a terminal. Async generators provide natural backpressure and streaming. A custom TUI framework (rather than a web app) gives developers a native terminal experience with zero context-switching from their IDE.
Chapter 2

Entry & Initialization Flow

How Claude Code boots — from the moment you type claude in your terminal to a fully interactive REPL.

Why parallel prefetching? MDM (Mobile Device Management) policy reads and macOS keychain lookups are slow I/O. By firing them before imports finish loading, the results are ready by the time they're needed — shaving hundreds of milliseconds off startup.
Why two modes? Interactive mode powers the TUI REPL for developers. Headless/SDK mode enables programmatic use — CI pipelines, editor extensions, and the Agent SDK — all sharing the same QueryEngine core.
Chapter 3

The Query Loop — Async Generator Pipeline

The beating heart of Claude Code. An async function* generator that streams events between the API, tool system, and UI — enabling natural backpressure and cancellation.

Why async generators? They provide lazy evaluation and natural backpressure. The UI pulls events at its own pace. If the user interrupts, the generator is simply returned/thrown — no complex cancellation logic needed. Each layer (queryModelWithStreaming → query → submitMessage → REPL) is a generator that yield*s from the layer below.
Why a while(true) loop? AI tool use is inherently iterative. The model generates a response, requests tools, gets results, then responds again. The loop continues until a terminal condition (stop, max turns, interrupt). This matches the agent pattern: think → act → observe → repeat.
Chapter 4

The Tool System

45+ tools with a unified interface, concurrent execution for read-only ops, serial execution for writes, and a streaming executor that starts tools before the model finishes responding.

Why partition into parallel vs serial? Read-only tools (Glob, Grep, FileRead) have no side effects — running them concurrently dramatically speeds up multi-file exploration. Write tools (Bash, Edit) must run serially to prevent race conditions on shared files.
Why a streaming executor? The model often requests multiple tools in a single response. The StreamingToolExecutor starts executing read-only tools as their JSON arrives in the stream — before the model even finishes talking. This overlaps network latency with tool execution, reducing total turn time.
Chapter 5

State Management

A hand-rolled, minimal store with immutable updates and selector-based React subscriptions — intentionally simpler than Redux or Zustand.

AppState Conversation State

~100+ fields: settings, model, permissions, messages, tasks, speculation, coordinator state. Fully immutable (DeepImmutable<T>). React components subscribe to slices via selectors — only re-render when their slice changes.

Bootstrap Process Singletons

Global mutable state for things that outlive any single conversation: session ID, cumulative token costs, registered hooks, telemetry providers. Deliberately kept small — the comment says "DO NOT ADD MORE STATE HERE".

Why hand-rolled? Simplicity

The store is ~30 lines. No middleware, no devtools, no action creators. Object.is skips no-op updates. useSyncExternalStore ensures tear-free reads in concurrent React. This is all a TUI needs.

Chapter 6

UI Layer — React in the Terminal

A custom Ink-inspired TUI framework rendering React components to terminal escape sequences. 146+ components power the interactive experience.

Why a custom TUI framework? Ink (the React terminal renderer) provides the base, but Claude Code needed custom features: streaming text rendering, focus management for permission dialogs, animated spinners, virtual message lists for performance, and terminal escape sequence handling (DEC codes for cursor control). Building on React means the same component model developers already know.
Chapter 7

Permission System

A layered security model ensuring the AI can't perform destructive actions without explicit consent. Static rules, hooks, classifiers, and interactive prompts form defense-in-depth.

Why defense-in-depth? An AI assistant with shell access is powerful but dangerous. No single layer is sufficient: static rules catch known patterns, hooks let organizations add custom policies, the classifier handles autonomous mode, and interactive dialogs are the final safety net. The "deny-first" evaluation order (deny rules checked before allow rules) ensures safety defaults.
Why a classifier in auto mode? When running non-interactively (CI, scripts), there's no human to approve. A lightweight model classifies each tool call as safe/unsafe, enabling autonomous operation with guardrails. Excess denials trigger a fallback to human prompting — the system fails safe.
Chapter 8

Hook Coordination System

An extensible event system that lets users, enterprises, and plugins inject logic at every stage of the tool lifecycle — from pre-execution to session end.

Why hooks instead of hardcoded checks? Different organizations have different security policies. A bank might require all file writes to be logged to an audit system. A startup might auto-approve everything in dev. Hooks make the permission and lifecycle system configurable without forking the codebase.
Why async hooks? Some validations (e.g., running a linter on a file after edit) are slow and shouldn't block the conversation. Async hooks run in the background and can "rewake" the agent with their results in a future turn — a form of eventual consistency for tool validation.
Chapter 9

Services Layer

Cross-cutting infrastructure: API communication, context management, protocol integrations, and observability.

API Claude API Client

services/api/claude.ts — Streaming requests via queryModelWithStreaming(). Supports Anthropic direct, AWS Bedrock, and GCP Vertex backends. Features: prompt caching with cache_control markers, exponential backoff retries (429/529), model fallback on failure, anti-distillation headers, and task budget configuration.

Why prompt caching? System prompts and conversation history are expensive to re-process. Cache breakpoints let the API skip re-encoding static content, reducing latency and cost by up to 90% on long conversations.

MCP Model Context Protocol

services/mcp/client.ts — Connects to external tool servers via stdio, SSE, StreamableHTTP, or WebSocket transports. Auto-generates MCPTool wrappers from server-advertised tool definitions. Handles OAuth token refresh, image resizing, and dynamic reconnection.

Why MCP? It's the open standard for AI-tool integration. Instead of hardcoding every possible tool, MCP lets Claude Code dynamically discover and use tools from any conforming server — databases, APIs, code analyzers, etc.

Compact Context Management

services/compact/ — Three strategies: autoCompact (summarize history when context fills), snipCompact (truncate old tool results), microCompact (deduplicate unchanged file reads). Monitors token usage via calculateTokenWarningState().

Why multiple strategies? Different situations need different compression. A 200-message conversation needs summarization. A single message with huge tool output needs truncation. Unchanged file re-reads just need deduplication. Layering these keeps context windows efficient.

OAuth Authentication

services/oauth/ — PKCE-based OAuth flow for claude.ai authentication. Spawns a local HTTP server to receive the redirect callback. Manages token refresh, scope negotiation, and keychain storage for persistent sessions.

LSP Language Server Protocol

services/lsp/ — Manages LSP server processes for code intelligence. LSPServerManager handles lifecycle, LSPClient speaks the protocol, LSPDiagnosticRegistry collects diagnostics. Powers the LSPTool for go-to-definition, diagnostics, etc.

Analytics Observability

services/analytics/ — GrowthBook for feature flags, OpenTelemetry for 1P event logging, Statsig for gates, Datadog for metrics. logEvent() is the central dispatch. Telemetry is opt-in and anonymized.

Chapter 10

Plugins & Skills

Two extensibility systems: Skills are prompt templates (Markdown with frontmatter) invokable as slash commands. Plugins are richer packages that bundle skills, hooks, MCP servers, and custom commands.

Why separate Skills from Plugins? Skills are lightweight — a single Markdown file is enough to create a reusable prompt template. Anyone can drop a .md file in ~/.claude/skills/ and it becomes a slash command. Plugins are for heavier integrations that need hooks, servers, or custom runtime behavior. This two-tier model serves both casual customization and deep extensibility.

Full Bootstrap Sequence

Every step from claude to first prompt, optimized for speed with parallel prefetching and lazy initialization.

// Startup timeline (simplified) Phase 1: Module Load (~50ms) ├── profileCheckpoint('main_tsx_entry') ├── startMdmRawRead() ← parallel I/O └── startKeychainPrefetch() ← parallel I/O Phase 2: init() (memoized, runs once) ├── enableConfigs() ├── applySafeConfigEnvironmentVariables() ├── setupGracefulShutdown() ├── Promise.all([1P logging, GrowthBook]) ← parallel ├── configureGlobalMTLS() + configureGlobalAgents() └── preconnectAnthropicApi() ← TCP/TLS warmup Phase 3: CLI Action Handler ├── ensureMdmSettingsLoaded() ├── loadRemoteManagedSettings() ├── loadPolicyLimits() ├── initializeGrowthBook() + fetchBootstrapData() ├── getInitialSettings() → effective config ├── initializeToolPermissionContext() ├── getTools() → 45+ tool instances ├── getMcpToolsCommandsAndResources() → MCP servers ├── initBuiltinPlugins() + initBundledSkills() ├── loadAllPluginsCacheOnly() ├── getCommands() → 100+ slash commands ├── processSetupHooks() ├── initializeLspServerManager() ├── showSetupScreens() → trust dialog if needed └── processSessionStartHooks() Phase 4: Launch ├── Interactive: createStore() → launchRepl() → <App><REPL /></App> └── Headless: new QueryEngine(config) → engine.submitMessage(prompt)
Why this ordering? Each phase is sequenced by dependency: you can't validate settings before configs are loaded, can't connect MCP servers before settings resolve, can't show trust dialogs before policy limits are known. Within each phase, independent operations run in parallel (MDM + keychain, 1P logging + GrowthBook). The result: first prompt readiness in under 2 seconds on most systems.
