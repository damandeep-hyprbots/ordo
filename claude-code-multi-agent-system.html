Chapter 1

Coordinator Mode

A pure orchestration mode where the coordinator cannot execute code directly. It delegates all work to async worker agents, keeping itself focused on synthesis, task planning, and communication with the user.

Why a pure orchestrator? By restricting the coordinator to only 4 delegation tools (Agent, TaskStop, SendMessage, SyntheticOutput), it is forced to think at the planning and synthesis level rather than getting lost in implementation details. Workers execute; the coordinator understands. This separation produces better task decomposition and prevents the coordinator from becoming a bottleneck by serializing tool calls through itself.

Config Enabling Coordinator Mode

Gated behind the COORDINATOR_MODE feature flag at build time, then activated at runtime via the environment variable:

export CLAUDE_CODE_COORDINATOR_MODE=1

The function isCoordinatorMode() in coordinatorMode.ts checks both. Session resume calls matchSessionMode() to flip the env var so a resumed coordinator session stays in coordinator mode.

Prompt System Prompt Structure

  • Defines coordinator as an orchestrator, not an executor
  • Documents the 4 available tools and their semantics
  • Explains <task-notification> XML format for worker results
  • Enforces parallelism: "Launch independent workers concurrently"
  • Synthesis rule: "Never write 'based on your findings'"
  • Continue vs spawn decision matrix based on context overlap

Context Worker Tool Context

getCoordinatorUserContext() builds a user context block that lists which tools workers have access to. In simple mode: Bash, Read, Edit. In full mode: the full ASYNC_AGENT_ALLOWED_TOOLS set minus internal-only tools (TeamCreate, TeamDelete, SendMessage, SyntheticOutput). Also advertises MCP servers and the scratchpad directory if enabled.

Phases The 4-Phase Workflow

  • Research -- Workers investigate (parallel, read-only)
  • Synthesis -- Coordinator reads findings, crafts implementation specs with file paths and line numbers
  • Implementation -- Workers make targeted changes per spec, commit
  • Verification -- Workers prove code works (not just "tests pass")
Chapter 2

Swarm System (Leader / Worker)

A multi-process team architecture where a leader and multiple workers run as separate Claude Code instances, coordinating through a file-based mailbox protocol for permission requests and message passing.

Why file-based mailbox? Each swarm member runs as an independent OS process (possibly in a different tmux pane or iTerm tab). There is no shared memory. The filesystem serves as a naturally persistent, lockable IPC channel. Workers write pending/{requestId}.json, the leader atomically resolves by moving it to resolved/, and workers poll the resolved directory. This avoids complex socket infrastructure while remaining robust across terminal restarts and reconnections.

Identity Leader Detection

The isTeamLeader() function checks two conditions:

const agentId = getAgentId() return !agentId || agentId === 'team-lead'

If CLAUDE_CODE_AGENT_ID is unset or equals 'team-lead', the process is the leader. Workers always have a distinct agent ID set.

Backends Three Execution Modes

  • TmuxBackend -- Spawns workers in tmux panes (cross-platform terminal multiplexer)
  • ITermBackend -- Uses native iTerm2 tabs (macOS, detected via isInITerm2())
  • InProcessBackend -- Workers run in the same Node process (no separate terminal, uses in-memory mailbox routing)

The registry.ts selects the appropriate backend based on environment detection.

Permission swarmWorkerHandler.ts

The worker permission flow in handleSwarmWorkerPermission():

  • Check isAgentSwarmsEnabled() && isSwarmWorker()
  • Try bash classifier auto-approval first
  • If unresolved, create SwarmPermissionRequest with tool details
  • Register callback BEFORE sending (avoids race condition)
  • Send request to leader via writeToMailbox()
  • Show visual pendingWorkerRequest indicator while waiting
  • Listen for abort signal to prevent hanging

Cleanup Resolution Lifecycle

Resolved permissions follow a strict lifecycle:

  • Pending files live in ~/.claude/teams/{team}/permissions/pending/
  • Resolved files move to .../permissions/resolved/
  • Workers poll with readResolvedPermission()
  • Old resolved files cleaned up after 1 hour via cleanupOldResolutions()
  • File operations use lockfile.lock() for atomic writes
Chapter 3

Fork Agents (runForkedAgent vs runAgent)

Two fundamentally different agent spawning strategies: runAgent creates a full-lifecycle independent agent, while runForkedAgent creates a lightweight child that shares the parent's prompt cache for near-zero startup cost.

Why fork instead of spawning a full agent? The Anthropic API caches request prefixes. When a fork child shares byte-identical system prompt, tools, model, and conversation history with its parent, the API serves the response from cache -- dramatically reducing latency and cost. The fork builds buildForkedMessages() with identical placeholder tool_result blocks for all tool_use calls, so only the per-child directive text block differs. This maximizes cache hit surface.
Why tag-based recursion prevention instead of a depth limit? Depth limits are fragile (what is the right number?) and require threading a counter through every call path. Instead, every fork child receives a message containing <fork-boilerplate> with strict execution rules. isInForkChild() simply scans the conversation for this tag. If present, the Agent tool refuses to fork again. This is binary, deterministic, and requires zero configuration.
// CacheSafeParams -- must be byte-identical between parent and fork for cache hits type CacheSafeParams = { systemPrompt: SystemPrompt // Must match parent userContext: { [k: string]: string } // Prepended to messages systemContext: { [k: string]: string }// Appended to system prompt toolUseContext: ToolUseContext // Contains tools, model, options forkContextMessages: Message[] // Parent conversation prefix }
Chapter 4

Agent Isolation (Worktrees)

Agents that modify files can operate in isolated git worktrees -- separate working copies of the same repository that prevent file conflicts between concurrent workers.

Why symlink node_modules? A full copy of node_modules can be hundreds of megabytes or even gigabytes. Since worktrees share the same repository contents at the git level, symlinking large directories avoids disk bloat while keeping the worktree functional. The symlink is created in symlinkDirectories() which handles ENOENT (source does not exist yet) and EEXIST (already linked) gracefully.
Why inject a worktree notice? Fork children inherit the parent's conversation context, which contains file paths pointing to the parent's working directory. buildWorktreeNotice() tells the child agent: "You are in an isolated worktree at {path}. Translate paths from the inherited context. Re-read files before editing." This prevents stale-read bugs and path confusion.
// Worktree notice injected into fork children operating in a worktree buildWorktreeNotice(parentCwd, worktreeCwd): "You've inherited the conversation context above from a parent agent working in ${parentCwd}. You are operating in an isolated git worktree at ${worktreeCwd} -- same repository, same relative file structure, separate working copy. Paths in the inherited context refer to the parent's working directory; translate them to your worktree root. Re-read files before editing if the parent may have modified them."

Agent State Sharing

How createSubagentContext() in forkedAgent.ts isolates mutable state while selectively sharing what is needed. Every field has explicit isolation semantics documented below.

readFileState Cloned from parent Cloned (overridable) Cloned (overridable) Prevents stale reads while preserving cache warm-start
abortController New child linked to parent New child (parent abort propagates) shareAbortController: true shares parent's Linked controller ensures parent abort kills child; shared for interactive agents that need UI

getAppState Wrapped with shouldAvoidPermissionPrompts: true Wrapped (avoids UI prompts) Parent's original (if sharing abortController) Background agents must not show permission UI; interactive agents can
messages Parent messages (overridable) Override with initial messages Override with initial messages Each agent builds its own conversation from its prompt

queryTracking.depth parent + 1 parent + 1 parent + 1 Analytics only -- NOT enforced as a recursion limit. New chainId per subagent.
localDenialTracking New instance New (accumulates across retries) Parent's (if sharing setAppState) Isolated denial counters for async agents whose setAppState is no-op
agentId createAgentId() Overridable Overridable Each subagent gets a unique ID for transcript tracking and cleanup
setAppStateForTasks Parent's (always shared) Always reaches root store Always reaches root store Task registration/kill must reach root even when setAppState is no-op, preventing zombie shell tasks
UI callbacks undefined undefined undefined Subagents cannot control parent UI (setToolJSX, setStreamMode, addNotification, etc.)
Why clone contentReplacementState instead of creating fresh? Cache-sharing forks process parent messages that contain parent tool_use_ids. A fresh replacement state would see them as unseen and make divergent replacement decisions, causing the wire prefix to differ and busting the prompt cache. A clone makes identical decisions, preserving the cache hit.
Why does setAppStateForTasks always reach the root store? Even when setAppState is a no-op for async agents, task registration and kill signals must still reach the global store. Otherwise, background shell processes spawned by async agents are never registered and never killed -- they become orphaned zombie processes (PPID=1). This is why setAppStateForTasks is separated from setAppState.
Chapter 6

Agent Color System

Each agent type gets assigned a unique color for visual differentiation in the terminal UI. The color system uses namespaced theme keys to prevent accidental cross-use between the main UI and subagent displays.

Why the _FOR_SUBAGENTS_ONLY suffix? The theme system is shared across the entire UI. Without namespacing, a developer might accidentally use theme.red (meant for error states) to color a subagent's output, or vice versa. The explicit suffix red_FOR_SUBAGENTS_ONLY makes it a type error to use agent colors outside the agent display system, and prevents main UI colors from leaking into agent rendering.
// agentColorManager.ts export const AGENT_COLOR_TO_THEME_COLOR = { red: 'red_FOR_SUBAGENTS_ONLY', blue: 'blue_FOR_SUBAGENTS_ONLY', green: 'green_FOR_SUBAGENTS_ONLY', yellow: 'yellow_FOR_SUBAGENTS_ONLY', purple: 'purple_FOR_SUBAGENTS_ONLY', orange: 'orange_FOR_SUBAGENTS_ONLY', pink: 'pink_FOR_SUBAGENTS_ONLY', cyan: 'cyan_FOR_SUBAGENTS_ONLY', } as const satisfies Record<AgentColorName, keyof Theme> // getAgentColor() returns undefined for 'general-purpose' agents export function getAgentColor(agentType: string): keyof Theme | undefined { if (agentType === 'general-purpose') return undefined const existing = getAgentColorMap().get(agentType) return existing ? AGENT_COLOR_TO_THEME_COLOR[existing] : undefined }
Chapter 7

SubagentStart / SubagentStop Hooks

Lifecycle hooks that fire before and after subagent query loops, enabling plugins and user-defined hooks to inject context, log activity, or perform cleanup when agents spawn and terminate.

Why auto-convert Stop to SubagentStop? Many existing hooks are configured for the Stop event. When these hooks fire within a subagent context (isAgent=true), the system automatically converts them to SubagentStop. This means SubagentStop hooks receive richer context (agent_id, transcript_path, last_assistant_message) without requiring every hook author to register for both event types. The conversion happens in stopHooks.ts where the hook event name is set to 'SubagentStop' when a subagentId is present.
Why provide the transcript path? The agent_transcript_path field in SubagentStop gives hook consumers access to the full sidechain transcript recorded during the agent's execution. This enables post-hoc analysis: a hook can read the transcript to extract metrics, verify quality, check for policy violations, or trigger follow-up actions -- all without needing to be wired into the agent's streaming pipeline.

Start executeSubagentStartHooks()

const hookInput: SubagentStartHookInput = { ...createBaseHookInput(undefined), hook_event_name: 'SubagentStart', agent_id: agentId, agent_type: agentType, }

Yields AggregatedHookResult with optional additionalContext that gets injected into the agent's context before its first API call.

Stop SubagentStop in stopHooks

// stopHooks.ts - auto-conversion const hookEvent = subagentId ? 'SubagentStop' : 'Stop' const hookInput = { hook_event_name: 'SubagentStop', stop_hook_active: stopHookActive, agent_id: subagentId, agent_transcript_path: getAgentTranscriptPath(subagentId), agent_type: agentType ?? '', last_assistant_message: lastAssistantText, }

The SubagentStop hook includes the full transcript path and the last assistant message text, enabling external tools to analyze the agent's complete execution.

Safety Trust Check

shouldSkipHookDueToTrust() prevents hooks from executing before the user has accepted the trust dialog. Historical vulnerability: SubagentStop hooks could execute when a subagent completed before the user decided whether to trust the workspace.

  • In non-interactive mode (SDK), trust is implicit -- always execute
  • In interactive mode, hooks are skipped until trust is established

Config Hook Registration

Hooks are registered via settings.json with event matchers:

  • SubagentStart -- matchQuery is hookInput.agent_type
  • SubagentStop -- matchQuery is hookInput.agent_type
  • Hooks run with TOOL_HOOK_EXECUTION_TIMEOUT_MS timeout
  • Plugin hooks loaded via loadPluginHooks.ts
  • Frontmatter hooks registered via registerFrontmatterHooks.ts
