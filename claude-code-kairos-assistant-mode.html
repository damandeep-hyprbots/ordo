Section 1

KAIROS Mode Overview

A build-time feature flag that transforms Claude Code from a request-response CLI into a persistent, always-on AI agent. The internal codename is tengu — every GrowthBook flag is prefixed tengu_.

Why "KAIROS"? In Greek, kairos means "the right moment." KAIROS mode is about the agent choosing when to act on its own — not waiting for a human to type a prompt. The tick loop gives the model continuous awareness of time passing, so it can decide the right moment to investigate, commit, push, or sleep.

BUILD-TIME Feature Flag

feature('KAIROS') is a Bun build-time constant. When false, entire KAIROS subsystems (Sleep, Brief, proactive prompts, cron tools) are dead-code-eliminated from the bundle. External/public builds ship with KAIROS = false.

RUNTIME GrowthBook Gates

Even when bundled, each subsystem has its own tengu_* GrowthBook flag as a kill-switch. This allows fleet-wide emergency disablement without a new client build.

  • tengu_kairos_brief — Brief tool
  • tengu_onyx_plover — Dream/autoDream
  • tengu_amber_quartz_disabled — Voice kill-switch
  • tengu_surreal_dali — Remote triggers
  • tengu_cobalt_frost — Voice Nova 3 STT
  • tengu_kairos_cron_config — Cron jitter tuning

PERSISTENT Always-On Agent

Unlike normal mode (user asks, model answers, loop ends), KAIROS runs an infinite tick loop. The session scheduler auto-starts without waiting for a user prompt. The model receives periodic <tick> heartbeat messages as synthetic user turns and must either do useful work or call Sleep.

CODENAME tengu

Internal codename for the KAIROS project. All GrowthBook flags use the tengu_ prefix. Analytics events use tengu_ prefixes (e.g., tengu_auto_dream_fired, tengu_brief_send). The name appears throughout the codebase in flag identifiers.

Section 2

Proactive Mode

The autonomous work system prompt, gated behind PROACTIVE || KAIROS. Defines the behavioral contract for a model running in an infinite tick loop without human prompting.

Why tick-based heartbeats? A persistent agent needs a way to "wake up" periodically even when no user input arrives. The <tick> messages arrive as synthetic user messages, giving the model awareness that time is passing. Each tick includes the user's local time so the model can reason about time-of-day without relying on external tool timestamps which may be in different timezones.

RULE Must Act or Sleep

On every tick, the model must either perform useful work (read files, run tests, make changes) or call the Sleep tool. Idle text output like "still waiting" or "nothing to do" is explicitly prohibited as token waste. This is the core behavioral contract of proactive mode.

PACING Cache-Aware Sleep

The prompt cache expires after 5 minutes of inactivity. The model must balance sleep duration against cache economics: sleeping <5 min preserves the cache (cheap wake-up), sleeping >5 min forces a full re-prefill (expensive wake-up). This is explicitly noted in the system prompt.

BEHAVIOR Focus-Gated Autonomy

The terminalFocus field tells the model whether the user's terminal window is focused or not. When unfocused, the model acts with maximum autonomy — make decisions, explore, commit, push. When focused, it becomes collaborative — surface choices, ask before large changes.

FORBIDDEN Anti-Narration

The prompt explicitly forbids step-by-step narration of tool calls. Text output is reserved for: decisions needing user input, high-level status at milestones ("PR created", "tests passing"), and errors/blockers. Everything else is noise the user can see in tool call output.

// System prompt excerpt (18_proactive_mode.md) "You are running autonomously. You will receive <tick> prompts that keep you alive between turns -- just treat them as 'you're awake, what now?'" "If you have nothing useful to do on a tick, you MUST call Sleep." "Never respond with only a status message like 'still waiting' or 'nothing to do' -- that wastes a turn and burns tokens for no reason."
Section 3

Sleep Tool

The model's mechanism to defer the next tick. Without it, the tick loop would force an API call every heartbeat — even when there is nothing to do. Sleep exists to make idling cheap.

Why not just use Bash(sleep ...)? A shell sleep holds an open child process and wastes a Bash slot in the concurrent tool executor. The Sleep tool is process-lightweight — it's a Node.js timer that doesn't consume a shell. It can also run concurrently with other tools without interference.

API COST Cache Economics

Each wake-up is an API call. The Anthropic prompt cache has a 5-minute TTL. If the model sleeps under 5 minutes, the next call gets a cache hit (cheap, fast). Over 5 minutes, the entire system prompt and conversation history must be re-prefilled (expensive). The Sleep tool prompt explicitly instructs the model to "balance accordingly."

OS LEVEL caffeinate Integration

On macOS, the preventSleep service spawns caffeinate -i -t 300 to create an idle-sleep power assertion. It uses reference counting: startPreventSleep() increments, stopPreventSleep() decrements. The process is restarted every 4 minutes as a self-healing mechanism — if Node is killed with SIGKILL, the orphaned caffeinate auto-exits after 5 minutes.

// tools/SleepTool/prompt.ts export const SLEEP_TOOL_PROMPT = `Wait for a specified duration. The user can interrupt the sleep at any time. Use this when you have nothing to do, or when you're waiting for something. You may receive <tick> prompts -- these are periodic check-ins. Look for useful work to do before sleeping. Prefer this over Bash(sleep ...) -- it doesn't hold a shell process. Each wake-up costs an API call, but the prompt cache expires after 5 minutes of inactivity -- balance accordingly.`
// services/preventSleep.ts const CAFFEINATE_TIMEOUT_SECONDS = 300 // 5 minutes const RESTART_INTERVAL_MS = 4 * 60 * 1000 // 4 min restart cycle spawn('caffeinate', ['-i', '-t', String(CAFFEINATE_TIMEOUT_SECONDS)], { stdio: 'ignore' }) // -i: prevent idle sleep (least aggressive -- display can still sleep) // -t: auto-exit timeout (self-healing if Node is SIGKILL'd)
Section 4

Dream System (autoDream)

Background memory consolidation. A forked subagent runs a reflective pass over memory files, synthesizing recent session transcripts into durable, well-organized memories for future sessions.

Why "Dream"? Like biological sleep, the dream system runs when the agent is not actively engaged with the user. It consolidates short-term session transcripts into long-term memory files — exactly the way sleep consolidates memories in the brain. The name is used throughout the codebase and in the user-facing UI ("DreamTask").
Why gate order matters: The three gates run cheapest-first. Gate 1 (time) costs one stat() call. Gate 2 (sessions) scans the transcript directory and is throttled to once per 10 minutes. Gate 3 (lock) involves a write. If gate 1 fails (not enough hours), gates 2 and 3 never run. Per-turn cost when enabled: one GrowthBook cache read + one stat.

Time: hours since lastConsolidatedAt >= minHours (one stat) // 2. Sessions: transcript count with mtime > lastConsolidatedAt >= minSessions // 3. Lock: no other process mid-consolidation

Section 5

Voice Mode

Push-to-talk speech-to-text via WebSocket streaming to Anthropic's voice_stream endpoint. Binary audio frames go up, TranscriptText and TranscriptEndpoint messages come down.

Why OAuth only? The voice_stream endpoint lives on Anthropic's API infrastructure and uses the same OAuth credentials as Claude Code's claude.ai integration. API keys authenticate against a different auth plane and cannot access this WebSocket endpoint. Bedrock and Vertex use AWS/GCP auth respectively. The code explicitly checks isAnthropicAuthEnabled() and that accessToken !== null.

INPUT Push-to-Talk

  • Default keybinding: SPACE
  • Hold to record, release to finalize
  • Auto-repeat key events reset a timer
  • No silence detection in push-to-talk mode — recording continues until key release
  • 16 kHz, 16-bit signed, mono PCM

BACKEND Audio Capture

  • macOS: Native NAPI module (cpal/CoreAudio)
  • Linux: NAPI -> arecord (ALSA) -> SoX fallback
  • Windows: NAPI only (no fallback)
  • NAPI loaded lazily on first keypress (~1-8s)
  • WSL2+WSLg works via PulseAudio RDP pipes

PROTOCOL WebSocket Wire Format

  • Up: Binary audio frames, KeepAlive JSON (8s), CloseStream JSON
  • Down: TranscriptText (interim), TranscriptEndpoint (final), TranscriptError
  • Fast path: post-CloseStream endpoint resolves in ~300ms
  • No-data timeout: 1.5s (server sent nothing)
  • Safety cap: 5s (last-resort if WS hangs)

KILL SWITCH tengu_amber_quartz_disabled

  • GrowthBook flag, default false
  • When flipped to true, voice mode is disabled fleet-wide
  • Fresh installs default to "not killed" (no GrowthBook cache yet)
  • Gates command registration, config UI, and runtime availability
  • Nova 3 STT gated separately: tengu_cobalt_frost

Section 6

Cron / Scheduled Agents

A two-mode scheduling system: durable tasks survive restarts (persisted to .claude/scheduled_tasks.json), session-only tasks die with the process. Recurring tasks re-fire; one-shots auto-delete.

Why per-project locks? Multiple Claude Code sessions can be open in the same project. Without a lock, the same cron task would fire in every session simultaneously. The lock file uses a session ID + PID scheme: the first session to acquire becomes the scheduler owner. If the owner crashes (dead PID), other sessions re-probe every 5 seconds and reclaim the lock.
Why jitter on :00 and :30? Humans round to half-hours when scheduling. "Check my PRs every hour" becomes 0 * * * *, and "remind me at 3pm" becomes 0 15 * * *. Without jitter, thousands of agents across the fleet would hit inference at the exact same second. The jitter system applies deterministic per-task hash-based delay: recurring tasks get forward delay (fire after the mark), one-shots get backward lead (fire before the mark, so they still feel "on time").
Config KeyDefaultPurpose
recurringFrac0.1Forward delay as fraction of inter-fire interval

Ops escape hatch: During an incident, ops can push tengu_kairos_cron_config with e.g. {oneShotMinuteMod: 15, oneShotMaxMs: 300000, oneShotFloorMs: 30000} to spread :00/:15/:30/:45 fires across a [t-5min, t-30s] window. Every task gets at least 30s of lead, so nobody lands on the exact wall-clock mark.
Section 7

Remote Triggers

Server-side trigger system via the claude.ai CCR API. Enables scheduled remote Claude Code agents that run on Anthropic's infrastructure, independent of the user's local machine.

Why in-process OAuth injection? The RemoteTrigger tool calls the API directly via axios with the OAuth bearer token added in-process. This is a deliberate security design: the token never appears in a shell command (unlike curl -H "Authorization: ..."), so it cannot be leaked through process listings, shell history, or tool call logs visible to the model.
Double gate: Remote triggers require both the tengu_surreal_dali GrowthBook flag AND the allow_remote_sessions organization policy. This means even if a user's GrowthBook enrollment is active, their organization admin must explicitly allow remote sessions. The tool's isEnabled() checks both at runtime.
// tools/RemoteTriggerTool/RemoteTriggerTool.ts isEnabled() { return ( getFeatureValue_CACHED_MAY_BE_STALE('tengu_surreal_dali', false) && isPolicyAllowed('allow_remote_sessions') ) } const TRIGGERS_BETA = 'ccr-triggers-2026-01-30'
Section 8

Brief Tool (SendUserMessage)

The model's primary output channel in KAIROS chat mode. Instead of writing assistant text directly, the model calls SendUserMessage(message, attachments, status). The model never sees its own message back — just gets a "Message delivered" acknowledgment.

Why a tool instead of plain text? In KAIROS mode, the model runs autonomously and most of its output is tool calls (file reads, bash commands, etc.). Plain text output gets mixed in with tool call chatter. By routing user-facing messages through a dedicated tool, the UI can cleanly separate "messages for the user" from "model working." The mapToolResultToToolResultBlockParam returns "Message delivered to user" — the model never sees its own message text echoed back, reducing token waste.

OUTPUT Message Schema

  • message: String with markdown support
  • attachments: Optional file paths (photos, screenshots, diffs, logs)
  • status: 'normal' (reply) or 'proactive' (unsolicited)

Attachments are resolved server-side: path, size, isImage, optional file_uuid for bridge uploads.

GATING Entitlement vs Activation

  • Entitlement (isBriefEntitled): Is the user allowed to use Brief? Checks KAIROS active, CLAUDE_CODE_BRIEF env, or tengu_kairos_brief GB flag.
  • Activation (isBriefEnabled): Is the tool active this session? Requires entitlement AND explicit opt-in (--brief, defaultView, /brief, or KAIROS mode).

The GB gate doubles as a kill-switch: flipping tengu_kairos_brief off disables the tool on the next 5-min refresh even for opted-in sessions.

ACK Token-Efficient Response

The tool result sent back to the model is just:

"Message delivered to user. (N attachments included)"

This prevents the model from seeing its own verbose output in the context window, saving tokens in long-running KAIROS sessions.

LEGACY Alias Support

The tool has an alias array for backward compatibility. Old transcripts may reference the legacy tool name. The aliases field on the tool definition ensures replayed sessions don't break when the UI encounters old tool_use blocks.
