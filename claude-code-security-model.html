Chapter 1

Defense in Depth

Claude Code has shell access, file write access, and API keys on your machine. A single security layer is never enough — here's how 8 independent layers stack together so any one failure doesn't compromise the system.

Why so many layers? An AI with shell access is as powerful as a junior developer with sudo. No single check is reliable alone — the model can be prompted adversarially, hooks can be misconfigured, classifiers can be fooled. Independent layers mean attackers must defeat all of them simultaneously.
Chapter 2

YOLO Classifier — Auto-Mode Permission AI

In auto-mode (--auto), there's no human to approve tool calls. A separate Claude model evaluates every action and returns a structured allow/deny decision.

Why a separate classifier model? The main Claude model has been prompted by the user and has established context that might bias its safety judgments. A fresh, independent model call with a minimal, sanitized view of the conversation makes manipulation much harder.

Chapter 3

OS-Level Sandbox

An external @anthropic-ai/sandbox-runtime package enforces filesystem and network restrictions at the OS level — independent of the application permission system.

Filesystem Write Controls

  • Always writable: . (cwd) and Claude temp dir
  • Always denied write: all settings.json files across all sources
  • Denied: managed-settings.d/, .claude/skills/
  • Additional dirs from --add-dir added to allow list

Bare-Repo Attack Mitigation

Blocks HEAD, objects, refs, hooks, config files in cwd. Prevents git's is_git_directory() from treating cwd as a bare repo — which would trigger core.fsmonitor hook execution.

Network Domain Allowlist

Allowed domains come from WebFetch permission rules (domain:* entries) + sandbox.network.allowedDomains. allowManagedSandboxDomainsOnly restricts to policy-only domains.

Platforms Supported

macOS, Linux, WSL2+. WSL1 explicitly not supported. enabledPlatforms setting allows gradual rollout (e.g., macOS-only for specific enterprise deployments like NVIDIA).

Auto-Allow Sandboxed Bash

autoAllowBashIfSandboxed=true (default) eliminates permission prompts for commands running inside the sandbox — the OS-level restriction is the safety boundary.

Settings Pinned Dual-Layer

Settings files are protected both at the application level (isClaudeSettingsPath()) AND injected as OS-level denyWrite entries in the sandbox config. Two independent barriers.

Why OS-level enforcement? Application-level permission checks can be bypassed if the model finds a creative tool sequence that the permission rules don't anticipate. An OS sandbox enforces constraints that no amount of clever prompting can circumvent.
Chapter 4

MDM — Enterprise Policy Enforcement

5-source priority chain for organizational policies, reading from OS-native management frameworks before any user settings.

Why use OS MDM instead of just config files? Config files can be modified by any process running as the user. macOS Managed Preferences and Windows HKLM require administrator/MDM-push to write — even a compromised user process can't override them. For enterprises managing thousands of developer machines, this is the standard distribution mechanism for software policies.
Chapter 5

Trust Dialog

First-launch safety checkpoint per project directory. Before Claude reads or writes anything, you confirm you trust the project.

Inspects What Gets Shown

  • MCP servers defined in project config
  • Hooks defined in project settings
  • Bash execution permissions from settings/skills
  • API key helpers, AWS/GCP commands
  • Dangerous environment variables

Yes, Trust Outcomes

Normal directory: persists hasTrustDialogAccepted: true for that path in project config.

Home directory (~): session-only trust — never persisted. Forces re-confirmation next launch.

No, Exit Hard Stop

Calls gracefulShutdownSync(1) — process exits. Claude Code will not operate in an untrusted directory. No partial initialization, no cleanup attempts.

Why per-directory trust? A malicious git repo could contain a .claude/settings.json with dangerous hooks pre-configured. The trust dialog forces a conscious decision before any of that configuration takes effect — preventing "git clone && claude" supply-chain attacks.
Chapter 6

Policy Limits

Organization-level feature restrictions fetched from Anthropic's API. Simple schema, careful fail-open/fail-closed design.

// Schema { restrictions: Record<string, { allowed: boolean }> } // Examples { restrictions: { "allow_product_feedback": { allowed: false }, "allow_external_mcp_servers": { allowed: false }, }} // Fail-open: absent key = allowed // ESSENTIAL_TRAFFIC_DENY_ON_MISS: // "allow_product_feedback" fails CLOSED in HIPAA mode

Storage ~/.claude/policy-limits.json

Cached with mode 0o600 (owner read/write only). ETag is SHA-256 of sorted restrictions for HTTP caching. Polled every hour in background.

Eligibility Who Gets Limits

First-party Anthropic URLs only. API key users always eligible. OAuth users must be on Team/Enterprise with claude_ai_inference scope.

HIPAA Mode Fail-Closed

In essential-traffic-only mode (HIPAA), allow_product_feedback and similar non-essential features fail CLOSED when cache is unavailable — prevents silent re-enablement.

Permission Rule Sources

8 sources, evaluated in order. Deny rules are checked first — any matching deny blocks the tool, no matter what allow rules exist.

PrioritySourceLocationWritable By

Why deny-first evaluation? The principle of least privilege. If any rule says "deny", it must win — allowing a tool should require explicit positive confirmation, not the absence of a denial. This prevents the common mistake of overly broad allow rules accidentally overriding targeted deny rules.
Shadowed rule detection warns when a specific allow rule (e.g., Bash(ls:*)) is unreachable because a broader deny rule (e.g., Bash) takes precedence. Prevents silent rule misconfiguration.
Chapter 8

Path Safety & TOCTOU Defense

Multi-layer path validation addressing shell expansion, symlinks, case sensitivity, and time-of-check vs time-of-use races.

Why block shell expansion syntax? TOCTOU (Time-Of-Check vs Time-Of-Use) race. The validator sees $TARGET_DIR/file.txt and might evaluate it as safe based on TARGET_DIR at check time. But when the shell executes the actual command, TARGET_DIR might resolve differently. Blocking $, %, and = prefix paths entirely eliminates this class of attacks.
Chapter 9

SSRF Guard

Server-Side Request Forgery prevention for HTTP hooks — blocks requests to internal infrastructure that could be used to pivot inside enterprise networks or access cloud metadata services.

Why DNS validation after resolution? A DNS rebinding attack registers a domain that resolves to a public IP during validation, then switches to an internal IP when actually requested. By resolving DNS ourselves and validating the IP before making the HTTP request, we close this gap.
Chapter 10

OAuth & Keychain Security

PKCE OAuth with hex-encoded credential storage — defeating plaintext monitoring tools while handling platform-specific buffer limitations.

PKCE OAuth Flow

  • randomBytes(32) → base64url for verifier and state
  • S256 challenge: SHA-256(verifier) → base64url
  • Local HTTP server for redirect callback
  • Manual fallback (user pastes code)

Hex Encoding CrowdStrike Defense

Credentials are Buffer.from(json).toString('hex') before keychain storage. Why: Defeats CrowdStrike-style security tools that grep process memory and stdin for plaintext tokens. Incident INC-3028. The hex string contains no recognizable token patterns.

4096-byte Guard fgets Overflow

macOS security CLI uses a 4096-byte fgets buffer. Payloads <4032 bytes go via stdin; larger payloads fall back to argv. Silent overflow would corrupt credentials without error.

Stale-While-Error Cache Resilience

On read failure (transient security process crash), previous valid cache entry is served. Prevents "Not logged in" flashes. Explicit logout sets data=null ensuring invalidation still works.

Session Lock Detection

SSH sessions have the keychain locked (not auto-unlocked at login). isMacOsKeychainLocked() checks exit code 36 from security show-keychain-info. Cached process-lifetime — ~27ms subprocess call.

Service Name com.anthropic.claudecode

Stored under this identifier in the macOS keychain. Uses the security CLI rather than Keychain APIs directly — works in all Terminal environments including SSH.

Chapter 11

Elicitation System

Structured user input requests from MCP servers — allowing tools to collect typed data from users with security hooks at every step.

Form Mode Typed Schema

MCP server sends a requestedSchema (JSON Schema). UI renders a typed form. Supported: string (email, uri, date, datetime), number, integer, boolean, enum (single-select), array (multi-select).

URL Mode Out-of-Band

Server sends a URL to open in browser. UI shows "waiting" state. After user completes action (e.g., OAuth), server sends ElicitationComplete notification. Natural language dates parsed via Haiku model call.

Security Hooks Before & After

executeElicitationHooks() before user sees dialog — hooks can programmatically accept/decline/cancel. executeElicitationResultHooks() after response — can modify or block the answer before sending to MCP server.

Chapter 12

Rate Limiting & Abuse Prevention

Response header-driven rate limit tracking with early warnings and classifier denial tracking to prevent denial-of-service feedback loops.

// Headers read from every API response anthropic-ratelimit-unified-status: "allowed" | "allowed_warning" | "rejected" anthropic-ratelimit-unified-representative-claim: "five_hour" | "seven_day" | "seven_day_opus" anthropic-ratelimit-unified-overage-status: overage quota state anthropic-ratelimit-unified-overage-disabled-reason: "out_of_credits" | "org_level_disabled_until" | ... // Early warning thresholds 5-hour window: warn at 90% utilization when ≤72% of window elapsed 7-day window: warn at 75%/50%/25% at corresponding time thresholds
Why classifier denial tracking? Without it, an adversarially-prompted model could get stuck in an infinite loop of attempting blocked actions. After 3 consecutive classifier denials or 20 total, the system falls back to human prompting — breaking the loop and alerting the user that something unusual is happening.
