Context Management

How Claude Code keeps conversations alive within a finite context window using five distinct compaction strategies and a content replacement system.

Section 1

The Context Window Problem

Claude models have a finite context window (typically 128K-200K tokens). Every user message, assistant response, tool call, and tool result accumulates. Long coding sessions with many file reads, shell commands, and edits fill it up fast. Without intervention, the conversation hits a wall.

The Accumulation Problem

A single FileRead of a large file can consume 5,000+ tokens. After 20-30 tool calls, the context is half full. A complex coding task can exhaust it in under an hour.

Five Strategies, One Goal

Each strategy addresses a different scenario: proactive vs. reactive, granular vs. wholesale, cache-preserving vs. cache-breaking. They coordinate through feature flags and mutual exclusion logic.

Plus Content Replacement

Orthogonal to compaction, content replacement enforces per-message budgets on individual tool results, persisting oversized outputs to disk and replacing them with previews before they even enter the context.

Why five strategies instead of one?

A single "summarize everything" approach wastes API calls and breaks prompt cache. Different strategies optimize for different conditions: warm cache (cached microcompact), cold cache (time-based microcompact), graceful degradation (auto-compact), last-resort recovery (reactive compact), and surgical precision (snip). The right tool for the right moment minimizes cost and latency while maximizing context quality.

Section 2

Decision Flow: Which Strategy Fires When

Before every API call, the system evaluates which (if any) context management action to take. The strategies are checked in priority order; most are mutually exclusive.

Section 3

Full Compact

The original context management strategy. A forked sub-agent reads the entire conversation and produces a rolling summary that replaces all prior messages.

Why a forked sub-agent?

Summarization is itself an LLM call that needs context space. By running it as a separate "compact" query, the system avoids recursive autocompact (the sub-agent's own context filling up and triggering another compaction), and can apply dedicated output token limits (COMPACT_MAX_OUTPUT_TOKENS). The forked agent also inherits separate abort/interrupt signals so users can cancel compaction without losing the conversation.

// services/compact/compact.ts — compaction entry point
const compactionResult = await compactConversation(
  messages,
  toolUseContext,
  cacheSafeParams,
  true,      // suppressUserQuestions for autocompact
  undefined, // no custom instructions
  true,      // isAutoCompact
  recompactionInfo,
);

// Post-compact: boundary marker + summary + hook results + attachments
// replace the entire message history
Section 4

Auto-Compact

Proactive compaction that fires automatically when token usage approaches the context window limit. Tries session memory compaction first, then falls back to full compact.

Constant Value Purpose
AUTOCOMPACT_BUFFER_TOKENS 13,000 Headroom below effective context window for autocompact trigger
WARNING_THRESHOLD_BUFFER_TOKENS 20,000 Buffer for UI warning display threshold
ERROR_THRESHOLD_BUFFER_TOKENS 20,000 Buffer for UI error display threshold
MANUAL_COMPACT_BUFFER_TOKENS 3,000 Blocking limit: prevents sending when nearly full
MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES 3 Circuit breaker: stops retrying after N consecutive failures

calculateTokenWarningState Return Values

function calculateTokenWarningState(tokenUsage: number, model: string): {
  percentLeft: number                // % remaining relative to threshold
  isAboveWarningThreshold: boolean   // tokens >= threshold - 20,000
  isAboveErrorThreshold: boolean     // tokens >= threshold - 20,000
  isAboveAutoCompactThreshold: boolean // autocompact enabled AND tokens >= threshold
  isAtBlockingLimit: boolean         // tokens >= effectiveWindow - 3,000
}
Why a circuit breaker?

Data showed 1,279 sessions had 50+ consecutive autocompact failures (up to 3,272!) in a single session, wasting approximately 250,000 API calls per day globally. The circuit breaker at MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES=3 stops this waste. If the context is irrecoverably over the limit (e.g., a single enormous tool result), retrying compaction is futile.

Environment Overrides

  • CLAUDE_CODE_AUTO_COMPACT_WINDOW — Caps effective context window size for testing
  • CLAUDE_AUTOCOMPACT_PCT_OVERRIDE — Set trigger as % of window
  • DISABLE_AUTO_COMPACT — Disable auto-compact (keeps manual /compact)
  • DISABLE_COMPACT — Disable all compaction

Mutual Exclusion

  • Suppressed when REACTIVE_COMPACT + tengu_cobalt_raccoon is on
  • Suppressed when CONTEXT_COLLAPSE is enabled
  • Never fires for querySource 'compact' or 'session_memory' (prevents recursion)
  • Never fires for 'marble_origami' (context agent)
Section 5

Microcompact (Two Sub-Modes)

Unlike full compact which summarizes the entire conversation, microcompact operates on individual tool results only. Two sub-modes handle different cache states.

Cached Microcompact CACHED_MICROCOMPACT flag

When the API prompt cache is warm, uses the cache_edits API to delete old tool results from the server-side cache without invalidating the cached prefix. The local messages are left unchanged; cache_reference and cache_edits blocks are injected at the API layer.

  • Tracks tool results via CachedMCState
  • Count-based trigger/keep thresholds from GrowthBook config
  • Only runs for main thread (not forked agents)
  • Pending cache edits consumed by API layer via consumePendingCacheEdits()

Time-Based Microcompact

When the gap since the last assistant message exceeds a configurable threshold (cache has expired), directly mutates message content to clear old tool results. Since the cache is cold anyway, there is no prefix to preserve.

  • Fires first and short-circuits (skips cached MC)
  • Replaces content with "[Old tool result content cleared]"
  • Resets cached MC state (stale tool IDs would cause errors)
  • Keeps last N results (floor at 1 to avoid empty context)
Why two sub-modes?

The API's prompt cache has a TTL. When the cache is warm, modifying message content would invalidate the entire cached prefix and force an expensive full re-read. The cache_edits API exists specifically to allow deletions without this penalty. But when the user has been idle long enough for the cache to expire, there is no prefix to protect -- so the simpler approach of directly clearing content is both safe and more effective (it shrinks what gets rewritten from scratch). Time-based fires first and short-circuits cached MC.

// Compactable tools (only these tool results are eligible for clearing):

])
Section 6

Reactive Compact

Instead of proactively compacting before hitting the limit, reactive compact waits for the API to actually reject the request with a 413 prompt_too_long error, then compacts in response.

How It Works

  • Feature flag: REACTIVE_COMPACT (ant-only)
  • GrowthBook gate: tengu_cobalt_raccoon
  • Suppresses proactive autocompact entirely when enabled
  • Waits for API 413 response, then calls tryReactiveCompact()
  • Also handles withheld media size errors

Mutual Exclusions

  • Mutually exclusive with Context Collapse
  • When active, shouldAutoCompact() returns false
  • Session memory compaction and manual /compact still work
  • isAutoCompactEnabled() stays true (reactive compact consults it)
Why wait for an error?

Client-side token estimation is approximate. The API has the true count. By waiting for the actual 413 error, reactive compact uses the server's authoritative measurement as the trigger, eliminating false positives from estimation drift. It also allows the context to fill more completely, maximizing the amount of conversation the model can "see" before summarization is forced. The 413 response includes a prompt_too_long token gap that tells the client exactly how many tokens to free.

Section 7

Snip (Surgical Message Removal)

Unlike summarization-based strategies, Snip lets the model itself surgically remove specific message ranges from its view. Messages are tagged with UUIDs, and the model calls SnipTool with the IDs of messages to remove.

How It Works

  • Feature flag: HISTORY_SNIP (ant-only)
  • Messages annotated with [id:<uuid>] tags
  • Model calls SnipTool with message UUIDs to remove
  • Not summarization -- messages are simply excluded
  • SNIP_NUDGE_TEXT meta-message encourages proactive use

Two Views of History

  • REPL (UI): Keeps full unsnipped history for scrollback
  • Model-facing: projectSnippedView() filters out snipped messages
  • On resume: applySnipRemovals() re-links parentUuid chains across gaps
  • Snip-freed tokens subtracted from autocompact calculations
Why model-driven removal?

The model knows which parts of the conversation are still relevant to the current task. A heuristic "oldest first" deletion might remove critical context. By letting the model choose which messages to snip, it can preserve important early decisions while removing completed subtasks. The SNIP_NUDGE_TEXT meta-message periodically reminds the model to use this capability proactively, and the freed tokens are subtracted from autocompact calculations so the two systems coordinate without racing.

Section 8

Content Replacement System

Orthogonal to conversation-level compaction, content replacement enforces a per-message budget on individual tool results. When results exceed the limit, they are persisted to disk and replaced with a preview stub.

Why freeze decisions?

Prompt caching is prefix-based: the API caches everything from the start of the prompt up to the last change. If a tool result was sent unreplaced on turn N, replacing it on turn N+1 would change the prefix and invalidate the entire cache. By freezing decisions in seenIds, the system guarantees that once a result is sent to the model, its representation never changes. On resume, reconstructContentReplacementState replays the exact same decisions from transcript records, producing byte-identical content for prompt cache hits.

Section 9

Session Memory Compaction

trySessionMemoryCompaction replaces traditional full compaction with the existing session memory as a ready-made summary. No separate LLM call needed -- the session memory system already maintains a running record of important context.

Configuration Thresholds

minTokens10,000Minimum tokens to keep after compaction
minTextBlockMessages5Minimum messages with text to keep
maxTokens40,000Hard cap on preserved tokens

Message Preservation Logic

  • Keeps messages from lastSummarizedMessageId forward
  • Expands backwards to meet minTokens and minTextBlockMessages
  • Stops expanding at maxTokens hard cap
  • adjustIndexToPreserveAPIInvariants: never splits tool_use/tool_result pairs or thinking block groups
  1. Auto-compact threshold is hit. autoCompactIfNeeded fires.
  2. trySessionMemoryCompaction is called first (before full compact fallback).
  3. Checks: is session memory non-empty? Does lastSummarizedMessageId exist in messages?
  4. calculateMessagesToKeepIndex determines which messages to preserve (respecting min/max thresholds).
  5. Session memory content is used as the summary (truncated if oversized). No LLM call.
  6. Boundary marker + summary + hook results + preserved messages replace the full history.
  7. If post-compact tokens still exceed threshold, returns null and full compact runs instead.
Why use session memory instead of a fresh summary?

Full compact requires a forked sub-agent LLM call that costs money and takes 5-15 seconds. Session memory is already being maintained as a side-effect of normal conversation -- it extracts key facts, decisions, and file locations continuously. Using it as the compaction summary is free (no API call), instant, and often higher quality because it is incrementally curated rather than generated in a single pass over a massive conversation dump. The system falls back to full compact only when session memory is empty or insufficient.

Section 10

VCR (Record/Replay) System

A cassette-style test recording system for API conversations. Records actual API calls to fixture files, then replays them deterministically in CI/test environments.

Dehydration (Record)

Normalizes environment-specific values so fixtures are portable:

  • Paths: /Users/me/project becomes [CWD]
  • Config: ~/.claude becomes [CONFIG_HOME]
  • UUIDs: a1b2c3d4-... becomes [UUID]
  • Timestamps: 2026-04-01T... becomes [TIMESTAMP]
  • Counters: num_files="42" becomes num_files="[NUM]"

Hydration (Replay)

Restores real values from the current environment:

  • [CWD] becomes current working directory
  • [CONFIG_HOME] becomes actual config path
  • Fresh randomUUID() for globally unique IDs
  • Costs re-added via addCachedCostToTotalSessionCost
  • Windows: forward/back slash normalization for cross-platform

Three VCR Modes

Non-Streaming withVCR

Records/replays a complete AssistantMessage[] array. Fixture keyed by SHA1 hash of dehydrated message contents. Used for standard request/response API calls.

Streaming withStreamingVCR

Wraps an AsyncGenerator of StreamEvent | AssistantMessage. Buffers all events during recording, then yields them synchronously on replay. Delegates to withVCR internally.

Token Count withTokenCountVCR

Records/replays token count API responses. Extra dehydration: CWD slugs ([CWD_SLUG]), UUIDs, and timestamps are normalized so fixture hashes remain stable across test runs.

// Fixture file naming: fixtures/{name}-{hash}.json
// Hash is SHA1 of dehydrated input, first 12 chars

// Dehydration example:
"/Users/me/project/src/foo.ts" -> "[CWD]/src/foo.ts"

// Windows cross-platform normalization:
"[CWD]\\src\\foo.ts" -> "[CWD]/src/foo.ts"
"[CWD]\\\\src\\\\foo.ts" -> "[CWD]/src/foo.ts"  // JSON-escaped
Why a VCR system?

API calls are expensive, non-deterministic, and slow. Tests need reproducible results without hitting production APIs. The VCR pattern (borrowed from Ruby's VCR gem) records real API interactions once, then replays the recorded responses in CI. Dehydration ensures fixtures work across machines, users, and operating systems. The VCR_RECORD=1 flag re-records when API behavior changes. Costs are re-added on replay so cost accounting tests stay accurate.
